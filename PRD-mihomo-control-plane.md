## 1. 产品概述

### 1.1 产品名称

Mihomo Control Plane

一个面向 `mihomo` 集中部署场景的 Web 管理台，提供配置维护、发布回滚、运行态观测和订阅集成能力。

### 1.2 一句话定义

不是另一个 Dashboard，而是一个面向集中部署 `mihomo` 的“配置控制台 + 发布系统 + 运行态观察台”。

### 1.3 背景

当前 `mihomo` 生态在“核心能力”上已经足够成熟，但在“集中部署后的可维护性”上存在明显缺口：

- 运行态面板大多只能看状态、切策略组、看日志，无法完整编辑节点、规则、策略组和 DNS。
- 桌面客户端虽然易用，但要求用户在本地跑核心，不适合统一部署在网关、NAS、VM、Linux 服务器上的场景。
- 直接维护 YAML 容易出错，尤其是规则顺序、策略组引用、节点命名、Provider 结构和发布回滚。
- 迁移自 Surge / OpenClash 的用户，最常见的流失原因不是核心功能不够，而是维护体验明显变差。

### 1.4 目标场景

- 家庭网络统一出口
- NAS / VM / Linux 服务器集中跑 `mihomo`
- 小团队共享代理出口
- 旁路由 / 透明网关场景
- 从 Surge / OpenClash 迁移到 `mihomo`，但不希望退回手写配置

## 2. 产品目标

### 2.1 核心目标

- 让用户不写 YAML，也能完成 80% 以上的 `mihomo` 日常维护工作。
- 让 `mihomo` 服务端具备接近 OpenClash 或商业代理客户端的管理体验。
- 让“配置编辑、发布、回滚、运行态观测”统一在一个 Web 入口完成。
- 保持 `mihomo` 核心为唯一真实数据面，不重新实现代理内核。

### 2.2 非目标

- 不替代 `mihomo` 核心。
- 不做本地桌面代理客户端。
- 不做 Surge `MITM / Rewrite / Script / Module` 兼容层。
- 不做 NAS / OpenWrt 的通用系统管理平台。
- 不优先覆盖所有实验性协议和极小众配置项。

## 3. 目标用户

### 3.1 用户画像

#### A. 家庭网络爱好者

- 在网关、NAS、虚拟机上统一部署 `mihomo`
- 希望全家设备共用一套规则和节点
- 不想每次通过 SSH 改配置

#### B. OpenClash / Surge 迁移用户

- 已习惯可视化维护节点、策略组和规则
- 接受 `mihomo` 作为核心
- 不能接受迁移后退化为手改 YAML

#### C. 运维型用户

- 需要配置校验、发布、回滚和审计
- 需要观察运行态、连接、规则命中和节点状态
- 可能维护多个实例

### 3.2 用户能力边界

- 理解“节点 / 策略组 / 规则 / Provider / 入口”的基本概念
- 不要求理解全部 YAML 细节
- 不要求有 Linux / SSH 经验

## 4. 关键问题定义

当前集中部署 `mihomo` 的用户，主要面临以下问题：

1. 配置维护分散
- 节点、策略组、规则、DNS、入口散落在 YAML 不同区域。
- 运行态切换和源配置维护不在一个工具里。

2. 改动风险高
- 引用关系难以检查。
- 配置错误可能导致服务无法启动。
- 缺乏发布历史和自动回滚。

3. 用户体验断层
- Dashboard 只能看，不能完整改。
- 桌面 GUI 能改，但核心不在服务端。
- 迁移自 Surge / OpenClash 的用户会明显感到降级。

4. 缺乏运行态闭环
- 配置编辑后，不容易验证某条规则是否真的命中。
- 很难快速判断“为什么某域名走了这个出口”。

## 5. 产品范围

### 5.1 MVP 范围

MVP 包含 6 个模块：

1. 节点管理
2. 策略组管理
3. 规则管理
4. Rule Provider 管理
5. 发布中心
6. 运行态观察

### 5.2 后续范围

- 订阅管理与外部订阅清洗
- 多实例管理
- 用户权限分级
- 审计日志与操作备注
- 配置漂移检测
- 模板市场 / 场景模板

## 6. 典型使用场景

### 场景 1: 新增一个代理节点

用户希望新增一个 `SS/Trojan/VLESS` 节点，并将其加入现有 `JP` 策略组。

成功标准：

- 无需手写 YAML
- 节点可在组中拖拽排序
- 发布前自动校验

### 场景 2: 某站点切换出口

用户希望将 `openai.com` 从 `JP` 改为 `US`，或改为 `MATCH`。

成功标准：

- 能搜索到现有规则
- 能调整规则顺序
- 发布后能看到命中结果

### 场景 3: 更新规则集

用户希望维护 `apple / cn / blocked` 等规则集。

成功标准：

- 可直接在线编辑或刷新
- 能看差异
- 不支持的规则类型会被预警

### 场景 4: 发布失败回滚

用户发布配置后发现 `mihomo -t` 或启动失败。

成功标准：

- 发布会被拒绝或自动回滚
- 上一个可用版本继续生效
- 可以查看失败原因

### 场景 5: 排查命中异常

用户希望确认某个域名为何走了特定出口。

成功标准：

- 能查看命中的规则
- 能查看所属策略组和最终出口
- 能查看相关连接日志

## 7. 功能需求

### 7.1 节点管理

#### 必须支持

- 新增、编辑、删除、复制节点
- 节点搜索、筛选、排序
- 支持主流协议：
  - `ss`
  - `trojan`
  - `vmess`
  - `vless`
  - `http`
  - `socks5`
  - `hysteria2`
  - `tuic`
- 节点字段按协议动态展示
- 节点健康检查与延迟测试

#### 应提供的体验

- 静态节点与订阅节点分开展示
- 支持批量导入
- 显示“该节点被哪些策略组引用”

### 7.2 策略组管理

#### 必须支持

- 支持 `select / fallback / url-test / load-balance / relay`
- 新增、编辑、删除策略组
- 拖拽调整组成员顺序
- 组引用节点、组引用组
- 显示当前策略组选中的真实出口

#### 约束检查

- 空策略组预警
- 引用不存在成员时阻止发布
- 递归引用预警

### 7.3 规则管理

#### 必须支持

- 新增、编辑、删除规则
- 支持规则类型：
  - `DOMAIN`
  - `DOMAIN-SUFFIX`
  - `DOMAIN-KEYWORD`
  - `IP-CIDR`
  - `IP-CIDR6`
  - `SRC-IP-CIDR`
  - `DST-PORT`
  - `PROCESS-NAME`
  - `GEOIP`
  - `RULE-SET`
  - `MATCH`
- 支持规则排序
- 支持按关键字搜索和过滤

#### 兼容性要求

- 对 `mihomo` 不支持的规则类型给出明确预警
- 对 Surge 风格规则给出识别与转换提示
- 清晰区分“规则内容错误”和“规则顺序风险”

### 7.4 Rule Provider 管理

#### 必须支持

- 支持 `file` 和 `http` 两类 Provider
- 支持在线编辑文本规则集
- 支持刷新远程 Provider
- 支持查看 Provider 差异和更新时间

#### 额外能力

- 自动检查 Provider 中是否包含不兼容规则
- 支持一键转换或清洗常见格式

### 7.5 系统配置管理

#### MVP 必须覆盖

- 端口
- `external-controller`
- `secret`
- 日志级别
- `allow-lan`
- `bind-address`
- `ipv6`
- `tun`
- DNS 主要配置
- `listeners`

#### 非 MVP 可暂缓

- 极少用的实验字段
- 完整 geodata 细节
- 所有插件和特殊传输参数

### 7.6 发布中心

#### 必须支持

- 草稿态与运行态分离
- 发布前执行配置校验
- 生成变更 diff
- 自动备份上一个版本
- 发布失败自动回滚
- 支持 reload / restart

#### 发布记录

- 记录发布时间
- 记录操作者
- 记录变更备注
- 记录是否成功

### 7.7 运行态观察

#### 必须支持

- 当前策略组状态
- 当前连接列表
- 实时日志
- 节点延迟
- Rule Provider 状态
- 规则命中查看

#### 关键能力

用户能够回答：

- 这个请求命中了哪条规则
- 它走了哪个策略组
- 最终从哪个节点出去

### 7.8 认证与权限

#### MVP

- 基础登录认证
- 管理员角色
- 只读角色

#### 后续

- IP 白名单
- SSO / OAuth
- 更细粒度 RBAC

## 9. 关键交互原则

### 9.1 双模式编辑

- 普通用户使用表单模式
- 高级用户可切换文本模式
- 两种模式之间保持同一份草稿状态

### 9.2 先校验再发布

- 所有配置改动默认只进入草稿
- 只有通过校验后才允许发布

### 9.3 引用透明

- 任意节点、策略组、规则都能看到“被谁引用”
- 删除前必须展示影响范围

### 9.4 风险优先

- 不兼容规则
- 悬空引用
- 空策略组
- 无默认兜底
- 关键端口冲突

这些风险必须在发布前明确提示。

## 10. 非功能需求

### 10.1 可靠性

- 面板故障不应影响已运行的 `mihomo`
- 发布失败必须可自动回滚
- 所有关键操作需幂等

### 10.2 安全性

- 敏感字段支持脱敏显示
- 日志中避免明文暴露密码和 token
- 发布记录保留操作者信息

### 10.3 性能

- 页面首屏加载时间小于 3 秒
- 发布校验结果 5 秒内返回
- 日志与连接页支持增量刷新

### 10.4 可维护性

- 配置结构和 `mihomo` 原生字段尽量一一对应
- 后端输出标准 `mihomo` YAML
- 升级面板不应破坏已有配置

## 11. 成功指标

### 11.1 MVP 指标

- 80% 常见维护操作无需 SSH
- 80% 常见维护操作无需手写 YAML
- 单次配置发布成功率 > 95%
- 规则命中排障时间降低 50%

### 11.2 用户体验指标

- 新用户 10 分钟内能完成：
  - 新增节点
  - 调整规则
  - 发布并验证
- 从 Surge / OpenClash 迁移的用户，不因维护体验下降而退回旧方案

## 12. 风险与约束

### 12.1 风险

- `mihomo` 配置面宽，字段众多，完整覆盖成本高
- 不同部署环境差异大，系统层配置很难完全统一
- 与运行态面板的职责边界容易混乱
- 与订阅清洗工具的边界需要明确

### 12.2 产品约束

- 必须以 `mihomo` 原生配置为最终真相
- 不引入与原生配置冲突的抽象层
- 不将系统管理和代理管理过度耦合

## 14. 验收标准

当以下条件同时成立时，MVP 可视为达标：

- 用户无需登录 SSH 即可完成节点、规则和策略组的常规维护
- 发布前有明确校验，发布失败会自动回滚
- 用户可以在 UI 中明确看到“规则命中 -> 策略组 -> 出口节点”的链路
- 运行态和源配置在同一个管理入口下闭环
- 普通用户可以在不理解完整 YAML 的前提下安全维护 `mihomo`
